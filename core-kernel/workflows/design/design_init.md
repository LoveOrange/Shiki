# Design Init

> Role: Architect. Analyze the Design Brief, expand the plan, and register targets. Do not write design body content.

## Load

- `features/{feature}/design_brief.md`
- `features/{feature}/_plan.md` bootstrap plan
- `features/{feature}/index.md`
- `core-kernel/runtime/design_contract.md`
- `core-kernel/templates/feature/_plan.md`
- `shiki_context/modules/{module}/index.md` when present

## Steps

1. Read the Design Brief and estimate size, scope, and risk.
2. Run the Design Contract Reuse Gate for planning scope: identify affected modules, existing specs, and why each planned item is required.
3. For structural changes, expand `_plan.md` into the smallest required model, persistence, ACL, component, entrance, flow, code, test, and merge tasks.
4. Use the `Target Outputs` table. Every feature plan `target` must be feature-root relative; `modules/{module}/...` means feature overlay and must not be written as `shiki_context/modules/{module}/...`.
5. Require every downstream Design leaf spec to fill `§0 Baseline Delta` with `reuse/add/extend/modify/deprecate` and `§0 Reuse Decision Gate`.
6. For tiny code-only fixes, keep the plan minimal and route to the smallest necessary tasks.
7. Create missing module overlay directories when needed.
8. Update feature index entries for generated targets.
9. Do not generate design specs or code body text.

## Output

- Update `features/{feature}/_plan.md`.
- Update `features/{feature}/index.md`.
- Do not update baseline module index in design_init; baseline index changes happen at merge.

## Verification

1. Plan shape matches the brief size.
2. Every contract path resolves.
3. Generated targets are feature-root relative and do not point to `shiki_context/modules/**`.
4. Feature index preserves Baseline Delta guidance for generated leaf specs.
5. Plan expansion includes only affected items, or returns `MANUAL_DECISION` when scope cannot be bounded.
6. No design body files are generated by this workflow.
7. README files are not used for runtime routing.
